1. 消息中间件与解耦（核心瓶颈）
demo演示：使用 ApplicationEvent（同步或默认异步）模拟 MQ，无法应对生产级的消息积压、持久化和批量消费。
上线方案：
1.替换为专业 MQ：引入RabbitMQ作为消息中枢。
2.消费隔离：行情 Topic（高吞吐）、订单回报 Topic、账户 Topic 使用独立的 Listener 和线程池，防止行情洪峰阻塞其他业务。
3.批量拉取：消费端启用批量拉取，减少网络 IO 次数。
4.背压机制：利用 MQ 的流控，当引擎处理不过来时，暂停消费（pause()），防止内存溢出。

2. 线程池与并发模型（性能瓶颈）
demo演示：线程池使用无界队列（LinkedBlockingQueue），高并发下可能导致 OOM。 
所有实例共用一个线程池，缺乏隔离。
上线方案：
1.有界队列 + 拒绝策略：使用 ArrayBlockingQueue 并设置合理容量，拒绝策略采用 DiscardOldestPolicy 或自定义降级（丢弃非关键策略）。
2.线程池隔离：
IO 密集型（DB/Redis 操作）：核心线程数 = CPU 核数 * 2。
CPU 密集型（表达式计算/DAG 执行）：核心线程数 = CPU 核数 + 1。
3.异步非阻塞：引入 CompletableFuture 编排任务，或使用 Reactor 响应式编程，减少线程上下文切换开销。

3. 订单队列与分发机制（可靠性瓶颈）
demo演示：
1.PriorityBlockingQueue 存在内存，重启即丢失订单。
2.调度器每秒拉取一次，吞吐量受限于单线程。
3.无失败重试机制。
上线方案：
1.持久化队列：使用 Redis Streams顺序消息 替代内存队列，支持持久化和优先级（通过自定义 Topic 分区实现）。
2.批量分发：改为 ScheduledExecutorService 批量拉取（如每次 500 条），使用 Netty 或 HTTP 连接池异步发送给交易接口。
3.失败重试与死信：
发送失败后，放入重试队列（延迟重试，如 1s, 5s, 30s）。
超过重试次数进入死信队列，人工介入。
4.幂等性：订单生成时增加全局唯一 ID（雪花算法），防止交易接口因超时重发导致重复下单。

4. 数据库与缓存策略（存储瓶颈）
demo演示：
1.JPA 使用 save() 单条插入，TPS 1000 时数据库连接池极易耗尽。
2.Flyway 与 Hibernate 双重管理 Schema 存在冲突风险（之前报错根源）。
上线方案：
1.批量写入：使用 JDBC Batch 或 MyBatis-Plus 批量插入 quote_order，每 500 条刷一次。
2.读写分离：strategy_def 和 strategy_instance 配置信息读多写少，使用 Caffeine 本地缓存 + Redis 二级缓存，减少 DB 压力。配置变更时通过 MQ 广播清理缓存。
3.冷热分离：订单表按日期分表（quote_order_202607），热数据仅保留近 3 天，历史数据归档到 ES 或 ClickHouse。
4.连接池调优：HikariCP 最大连接数设置为 (CPU核数 * 2) + 磁盘数。

5. 流程执行引擎（计算效率瓶颈）
demo演示：
1.每次执行都解析 flow_json（虽已缓存，但表达式每次用 Aviator 重新执行）。
2.MarketDataNode 从静态 Map 取数据，分布式下无法共享。
上线方案：
1.表达式预编译：将 Aviator 表达式预编译为 Expression 对象，缓存到 ExecutionPlan 中，执行时直接调用 execute(varMap)。 --- 已优化
2.行情数据集中获取：MarketDataNode 改为批量预加载（一次行情事件，预先将涉及的所有标的行情从 Redis Pipeline 取出），避免多次网络请求。
3.上下文传递优化：使用 ThreadLocal 或对象池（Recycler）复用 Map 对象，减少 GC 压力。
4.分布式缓存：行情缓存存放在 Redis，使用 HSET 存储行情字段，订阅 Redis 的 Keyspace Notifications 或 MQ 更新本地缓存。

6. 集群部署与状态同步（高可用瓶颈）
demo演示：单机部署，存在单点故障；缓存和队列都是内存态，无法水平扩展。
上线方案：
1.无状态化设计：将 ExecutionPlan 缓存放到 Redis（序列化为 JSON），所有节点共享。
2.分布式锁：确保同一个策略实例不会在多个节点同时执行（如使用 Redisson 的 RLock，key = instance_id）。
3.优雅启停：注册 ShutdownHook，应用关闭时将队列中剩余的订单持久化到 DB，并暂停 MQ 消费。
4.容器化与编排：使用 Docker + K8s 部署，配置存活探针（/actuator/health）和就绪探针，实现滚动更新。

7. 可观测性（运维排查瓶颈）
demo演示：只有控制台日志，生产环境无法定位慢节点。
上线方案：
1.Metrics（指标）：集成 Micrometer + Prometheus，监控：
引擎执行耗时（分位数 P99, P95）。
队列深度与积压量。
线程池活跃度、拒绝次数。
DB 连接池占用率。
2.Tracing（链路）：集成 SkyWalking 或 Zipkin，给每个行情事件注入 TraceId，贯穿接收 -> 计算 -> 订单生成 -> 推送全流程。
3.Logging（日志）：日志结构化（输出 JSON 格式），包含 instance_id, symbol, trace_id，接入 ELK。
4.告警规则：队列深度 > 10000 触发告警；执行耗时 > 500ms 触发告警。

8. 外部接口容错（稳定性瓶颈）
demo演示：TradingClient 模拟无超时处理，真实网络环境会超时或闪断。
上线方案：
1.超时与熔断：引入 Resilience4j，配置超时（Timeout）、熔断（CircuitBreaker）和限流（RateLimiter）。
2.异步 HTTP 客户端：替换同步 RestTemplate 为 Apache HttpClient 5 或 OkHttp 连接池，启用 Keep-Alive。
3.重试风暴避免：交易接口出现故障时，采用指数退避（Exponential Backoff）重试，并随机抖动（Jitter），防止雪崩。


9. 安全性加固
1.配置加密：application.yml 中的数据库密码使用 Jasypt 加密。 
2.接口鉴权：前端 API 和内部接口（如 /simulate）增加 Token 校验（Spring Security OAuth2），或移除测试接口。
3.防注入：Aviator 表达式引擎启用 沙箱模式，禁止访问 System、Runtime 等危险类。

10. 性能压测预估目标
1.延迟：从收到行情到订单入队，P99 < 50ms。
2.吞吐量：单机支持 10000 TPS（实例触发 + 订单生成）。
3.可用性：集群部署下，单节点宕机不影响服务，RTO < 30s。
